Detailed Description

Provide an array of bytes.

mdtFrame is based on Qt's QByteArray, that provide a array of bytes. To store data, the QByteArray's methods append() can be used.
For ASCII frames reception, the end is a special char/string on most case. mdtFrame provide the functions putUntil() for that purpose.
During data reception, for exemple from serial port, the best way to check if more data can be stored is to use bytesToStore(). It returns the remain capacity of the frame, and 0 if the end of frame (EOF) condition was reached. To know if the frame is complete (i.e. end of frame condition ok), use isComplete()
Here are the states of a frame with return values of both functions:

StatebytesToStore()isComplete()
Frame is not full, and EOF condition not reachedremainCapacity()false
Frame is full, and EOF condition not reached0false
Frame is not full, and EOF condition reached0true
Frame is full, and EOF condition reached0true

At this state, whenn bytesToStore() returns 0 and isComplete() is true, the received frame is ok. The next step is to decode it, according to the used protocol (modbus, ...).
Note that these states are only set when using putUntil() and putData() functions.

int mdtFrame::putUntil ( const char *  data,
char  token,
int  maxLen 
)

Put data until EOF token was reached.

This function is usefull for ASCII frames reception. Whenn token is reached, the EOF condition will become true (the EOF char is not stored).
When the EOF condition was reached, this function does not store any more data.
Note that this function is slower than QByteArray's append() method, because it compares each byte with the token.

Parameters
dataPointer to source data
tokenToken that represents the EOF (End Of Frame)
maxLenLength of the array pointed by data
Returns
The number of bytes really stored
Precondition
The data pointer must be valid, and not be the internal pointer returned by data().

void mdtFrame::setWaitAnAnswer ( bool  wait)

Tell a write/read thread that an answer is needed after a write frame.

In some case, typically USB, the host must allways init a transaction, for read and for write. For such case, only one thread manages the write/read process:

  • Wait until a frame must be sent (can be bypassed with a flag)
  • Init a write transaction, wait until transaction is finish
  • Init a read transaction, wait until something was readen. The last step can be a problem if we have to write a frame (f.ex. a query), but not need an answer. After that, we have to send a second query, but need wait until data was received.

To solve this problem, we must tell the thread if it must init a read transaction after a frame was written (request) or not, and this is the goal of this method.
