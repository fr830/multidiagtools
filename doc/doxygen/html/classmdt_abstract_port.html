void mdtAbstractPort::close ( ) [virtual]

Close the port.

This method must be re-implemented in subclass.
To handle the port correctly, the subclass method must:

  • Lock the mutex with lockMutex()
  • Do the specific work
  • Call this close method (with mdtAbstractPort::close() ). At this last step, the queues will be deleted, and mutex unocked.

Reimplemented in mdtPortPosix, mdtUsbtmcPort, mdtTcpSocket, and mdtSerialPortPosix.

bool mdtAbstractPort::open ( mdtPortConfig &  cfg) [virtual]

Open the port.

This method must be re-implemented in subclass.
To handle the port correctly, the subclass method must:

  • Close previous opened ressource
  • Lock the mutex with lockMutex()
  • Do the specific work
  • Set the read/write timeouts. See the mdtPortConfig to know how to get these timeouts.
  • Call this open method (with mdtAbstractPort::open() ). At this last step, the queues will be initialized, and mutex unocked.
    Returns:
    True on successfull configuration and open port
    See also:
    mdtPortConfig

Reimplemented in mdtPortPosix, mdtUsbtmcPort, and mdtTcpSocket.

bool mdtAbstractPort::waitForReadyRead ( int  msecs)

Wait until data is available at device.

This method calls setReadTimeout() and waitForReadyRead() (it is a little bit slower than setting timeout one time, and call waitForReadyRead() ).
Note that the reader thread will call waitForReadyRead() without argument.
Note: this method is called from thread , and should not be used directly
Mutex: see waitForReadyRead()

Returns:
False on error, in this case, the reader thread will be stopped.
